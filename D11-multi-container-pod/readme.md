## explain multi container pod
👉 Multiple containers running inside the SAME pod

Which means they:
    ✔ Share network namespace (same IP / localhost)
    ✔ Share storage volumes
    ✔ Share lifecycle (scheduled together, die together)

But each container still:
    ✔ Has its own filesystem
    ✔ Own process space

Think of it as cooperating containers.

🔥 Why do we even need multi-container pods?

    Because sometimes containers need tight coupling:
        ✔ Work together
        ✔ Communicate via localhost
        ✔ Share files
        ✔ Start/stop together

    If containers are independent → separate pods
    If containers are tightly linked → same pod

✅ Core Concept (VERY IMPORTANT)

    Inside a pod:

        👉 Containers behave like processes inside one machine

    Example:
        Container A → localhost:8080
        Container B → can call localhost:8080

    No service needed.

Classic Example:
    apiVersion: v1
    kind: Pod
    metadata:
    name: multi-container-demo
    spec:
    containers:
    - name: app
        image: nginx

    - name: sidecar
        image: busybox
        command: ["sh", "-c", "while true; do echo hello; sleep 5; done"]

    Two containers:
        ✔ nginx
        ✔ busybox

    Same pod, same IP.

🔥 Are there “types” of multi-container pods?
    Kubernetes doesn’t officially label them as types, but architectural patterns exist.
    This is what interviews & CKA care about.

    ✅ 1️⃣ Sidecar Pattern (Most Important ⭐)

        Most common design.
            👉 Helper container supporting main container.

        Used for:
            ✔ Logging
            ✔ Monitoring
            ✔ Proxy
            ✔ Config reload
            ✔ Service mesh

        Pod:
            ✔ Main app writes logs
            ✔ Sidecar ships logs (Fluentd)

        Containers:
            ✔ app container
            ✔ log-agent container

    ✅ 2️⃣ Ambassador Pattern

        👉 Proxy container acting as intermediary.

        Used for:
            ✔ External service access
            ✔ DB proxy
            ✔ TLS termination

        ✔ Example
            App → talks to localhost
            Ambassador → forwards to external DB

        Why useful?
            ✔ App doesn’t know external details
            ✔ Proxy handles complexity

    ✅ 3️⃣ Adapter Pattern

        👉 Container transforms data for main container.

        Used for:
            ✔ Metrics conversion
            ✔ Log format conversion
            ✔ Protocol translation

        ✔ Example
            App → emits raw metrics
            Adapter → converts to Prometheus format

❌ “Containers are isolated”

    Not fully.

        ✔ Process isolated ✅
        ✔ Network shared ❗
        ✔ Volume shared ❗
    
❌ “Use multi-container for microservices”

    Usually wrong.

        Microservices → separate pods.

        Multi-container → tightly coupled helpers.

🔥 When SHOULD you use multi-container pods?

    ✔ Sidecar utilities
    ✔ Proxies
    ✔ Init logic helpers
    ✔ Data transformation helpers

🔥 When should you NOT?

    ✔ Independent scaling needed
    ✔ Different lifecycle
    ✔ Separate failure domains

    Because:

    👉 Pod = smallest deployment unit
    You scale pod, not container.

🔥 Ultimate Mental Model 😈

    Multi-container pod = Cooperating processes
    NOT independent applications.

## Examples:

✅ 1️⃣ Sidecar Pattern — Log Generator + Log Reader (sidecar-pattern.yaml)

    👉 One container writes logs
    👉 Sidecar reads same logs

    This clearly shows:

    ✔ Shared volume
    ✔ Shared lifecycle
    ✔ Sidecar usefulness

    🔹 Apply
            kubectl apply -fsidecar-pattern.yaml
    🔹 Observe
            kubectl logs sidecar-demo -c log-reader

        ✔ You’ll see logs generated by other container
        ✔ Clean sidecar behavior

✅ 2️⃣ Ambassador Pattern — Local Proxy to External Service

    👉 App talks to localhost
    👉 Ambassador forwards traffic

    We’ll simulate proxy using socat.

    Step 1 — Run backend service
        kubectl apply -f amb-backend-service.yaml

        kubectl expose pod backend --port=80
    
    Step 2 — Ambassador Pod (ambassador-pattern.yaml)
        kubectl apply -f ambassador-pattern.yaml

    🔹 Observe
        kubectl logs ambassador-demo -c app

        ✔ App hits localhost
        ✔ Ambassador forwards to backend service
        ✔ Classic proxy behavior

✅ 3️⃣ Adapter Pattern — Transform Data (adapter-pattern.yaml)

    👉 App generates raw data
    👉 Adapter transforms it

    Simple & visual.

    step 1 : apply 
        kubectl apply -f adapter-pattern.yaml

    🔹 Observe
            kubectl exec adapter-demo -c adapter -- cat /data/converted.txt

        ✔ Data transformed
        ✔ Adapter role clear

🔥 Mental Model Reinforcement 😈
| Pattern    | Purpose               |
| ---------- | --------------------- |
| Sidecar    | Assist main container |
| Ambassador | Proxy / connectivity  |
| Adapter    | Data transformation   |
